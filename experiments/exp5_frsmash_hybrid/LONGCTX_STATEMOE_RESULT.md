# 超长上下文验证: Dense State-MoE 终于赢 (seq4096, state 容量是长程瓶颈)

用户提议: 用梯度累计在长序列验证 state 容量是否变瓶颈。seq512 不是; seq4096 呢?
梯度累计(micro_bs=1, accum=8, eff_bs=8) 在 4090 上跑 seq4096。

## 结果 (~82M, d512/L8, seq4096, 200步, 梯度累计)

| 配置 | val ppl(4096) | val ppl(trunc1024) | Δppl | 速度 |
|------|---------------|---------------------|------|------|
| vanilla FRSMASH | 82.36 | 82.81 | +0.44 | 95k tok/s |
| **dense State-MoE(10×)** | **81.17** | 81.32 | +0.16 | 60k tok/s |

**dense State-MoE 在 seq4096 比 vanilla 好 1.4% ppl(81.17 vs 82.36)!**

## 完整对照: state 容量是"上下文长度相关"的瓶颈
| seq | vanilla ppl | dense ppl | dense 效果 |
|-----|-------------|-----------|-----------|
| 512(1500步) | 36.99 | 37.66 | **-1.8%(更差)** |
| **4096(200步)** | 82.36 | **81.17** | **+1.4%(更好!)** |

- **seq512**: state 非瓶颈 → 10× 状态冗余 → 不提升。
- **seq4096**: state 变瓶颈 → 10× 状态有用 → dense 胜 1.4%。
- **结论: state 容量是上下文长度相关的瓶颈。短上下文不是, 超长上下文是。**

## 诚实细节
- **1.4% 是 modest**, 且 dense 代价 36% 速度(60k vs 95k tok/s, 10× SlowMemory 递归)。
- **Δppl**: dense 的 Δppl 更小(+0.16 vs +0.44)——说明 dense 的状态在 trunc1024 时就已
  捕获更多信息(81.32 < vanilla 的 82.81 全 4096), 所以看更多增益不大。dense 的状态更高效。
- 外推: **seq 8k+ 可能差距更大**(state 瓶颈更严重), 本设置(seq4096)已确认方向。

## 诊断闭环(loop 10 → 26 → 本)
1. **loop10 masked State-MoE**: 输入饥饿(状态 1/19) → 诊断: 稀疏版技术错误。
2. **loop26 dense State-MoE @seq512**: 全强度状态(1770)但 state 非瓶颈 → 不提升。
3. **本节 dense State-MoE @seq4096**: 全强度 + state 变瓶颈 → **提升 1.4%**。

**三步闭环确认: (a) masked 稀疏是技术错误(可修); (b) dense 修好后全强度; (c) 全强度 10× 状态在长上下文(4096)有用, 短上下文(512)无用。**

## 一句话
> **dense State-MoE(10× 全强度状态)在 seq4096 终于赢 vanilla 1.4%, 证实"state 容量是长程上下文的瓶颈"**——
> seq512 不是瓶颈(10× 冗余), seq4096 是瓶颈(10× 有用)。**state 容量瓶颈是上下文长度相关的**;
> 越长的上下文, 10× 状态越有价值。梯度累计让单卡也能验证这个长程结论。
