# Dense State-MoE (全强度 10× 状态): 诊断正确, 但 state 不是瓶颈 (诚实终判)

诊断(loop10)说 masked 版因输入饥饿失效(状态 1/19 强度)。本版 dense: 每专家看全序列,
门控混合 → 全强度状态。验证诊断 + 测能否提升。

## 结果 (~82M, d512/L8, seq512, 1500步)

| 配置 | 参数 | val ppl | tok/s | 单状态范数 |
|------|------|---------|-------|-----------|
| vanilla FRSMASH | 80.3M | **36.99** | 85k | 1052 |
| masked StateMoE(loop10) | 82.1M | 37.38 | 76k | **56(饥饿)** |
| **dense StateMoE(本)** | 82.1M | 37.66 | 75k | **1770(全强度!)** |

## 两层诚实结论

### ✅ 诊断正确: dense 版状态确实全强度了
dense 版每专家状态范数 **1770**(远超 vanilla 1052, 更远超 masked 版 56)。**输入饥饿被修好了**——
每专家看全序列, 状态充分累积。诊断"masked 版失败因饥饿"成立。

### ❌ 但 state 容量不是 FRSMASH 的瓶颈
**即便状态全强度且 10×(1770 vs 1052), ppl 仍不提升(37.66 vs 36.99, 反而略差 1.8%)。**
=> **SlowMemory 的状态容量根本不是 FRSMASH 的瓶颈。** 给它 10× 全强度状态也白搭——
SSM 骨干(8层)+ GLA recall 提供的状态已经够用, 多给的 10× SlowMemory 状态是冗余容量, 无处发挥。

## State-MoE 这条线的最终终判(两版都败, 原因不同)
| 版本 | 失败原因 |
|------|----------|
| masked(稀疏) | 输入饥饿(状态 1/19 强度)——技术错误 |
| **dense(全强度)** | **state 非瓶颈——根本性无用** |

**稀疏版是"做错了"(可修), dense 版是"做对了但没用"(state 非瓶颈)。** 修好技术错误后,
真正的结论浮现:**FRSMASH 在此规模不缺状态容量, 缺的是别的东西(数据/规模/SSM 主干宽度/训练步数)。**

## 诚实的外推边界
- "10× 状态有用"的前提是 **state 是瓶颈**。本实验(seq512/80M/1500步)state 非瓶颈, 所以 10× 无效。
- 可能在**超长上下文(8k-32k token)**下, 状态容量变瓶颈, dense State-MoE 才有用武之地。本设置测不到。
- 即便那样, dense 10× 算力的代价也得权衡。

## 一句话
> Dense State-MoE **证实了诊断(全强度状态达成, 范数 1770), 但揭示更深真相: state 容量不是 FRSMASH 的瓶颈**——
> 10× 全强度状态也换不来 ppl 提升(反而略差)。State-MoE 这条线两版都败: 稀疏版技术错(饥饿),
> dense 版做对了但根本无用(非瓶颈)。**FRSMASH 的提升空间不在状态容量, 在别处。**
