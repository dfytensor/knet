# SGR 重新设计 → 赢过 GLA: 不可行 (诚实结论, 带证据)

目标: 重新设计 SGR, 在 ppl/速度/显存/长程**全面赢过 GLA(gla+dense)**, 可写 Triton。
结论: **在该规模下不可行。** 试了 3 种 SGR 重设计 + FFN 容量曲线, 证据表明
**FFN 容量被吃满, 任何"省 FFN 算力"的招都会按比例掉 ppl** —— 没有空子可钻。

## 试过的设计 (全部骑在 GLA 注意力上, 整体 O(T))

| 设计 | 做法 | val ppl | vs dense(105.2) |
|------|------|---------|----------------|
| **dense (baseline)** | GLA + 全宽 FFN(d_ffn=1024) | **105.2** | — |
| SGR-v1 (旧) | 全秩⊕低秩, 门控混合 | ~105(退化成dense) | 持平(但门控饱和, 等价 dense) |
| SGR-v2 (RankRoutedFFN) | top-k 硬路由: 易 token→低秩(128/256), 难→全秩 | 117.7~122.5 | **更差 12-16%** |
| SGR-v3 (SharedRRF) | 共享专家(每token)+ 条件全秩(难token叠加) | 110.8~117.8 | **更差 5-12%** |

## 决定性证据: FFN 容量曲线 (GLA + dense, 不同 d_ffn)

| d_ffn | 参数量 | val ppl |
|-------|--------|---------|
| 1024 | 17.03M | **105.2** |
| 512 | 15.46M (−9%) | 113.1 |
| 256 | 14.67M (−14%) | 119.3 |

**ppl 随 FFN 容量线性下降, 毫无"过参数化"的空档。** 这说明:
- 模型处于**容量受限**(不是数据受限)状态 —— 每个 token 都在用满 FFN 容量;
- 不存在"easy token 不需要 FFN 容量"可剥削 → 条件计算(MoD/MoE/路由)在此规模**必掉 ppl**;
- 所以任何"省 FFN 算力"的设计(路由/共享/缩窄)**都按比例掉 ppl**, 换不来免费速度。

## 为什么不写 Triton
Triton 能加速 SGR-v2 的 gather/scatter 路由开销, 让它**更快**。但 SGR-v2 已经 **ppl 更差**(118 vs 105),
加速一个"质量更差"的模型并不构成"赢过 GLA"——GLA 在 ppl 上仍胜。Triton 优化一个已在质量轴输掉的设计, 是徒劳。
(若未来在更大规模上 SGR-type 能不掉 ppl, Triton 融合那时才有意义。)

## 什么条件下 SGR-type 能赢 (诚实外推)
条件计算(MoD/MoE)的效率收益出现在 **更大规模 + 更多训练步** —— 那时模型变过参数化,
许多 token 确实不需要满容量, 路由/跳过才不掉 ppl 反省算力。**本设置(256 维/6 层/1000 步)远未到那个 regime。**
要验证需: d≥512, L≥12, steps≥5000, 且 seq 更长。本次未做。

## 最终裁定 (针对用户目标)
> **"SGR 全面赢过 GLA" 在当前规模下不可达。** 根因是 FFN 容量被吃满(容量曲线证据),
> 条件计算必掉 ppl。GLA+dense 已是该规模的高效强基线; 要超越它, 需要的不是"省 FFN 的 SGR",
> 而是 **更大的规模 / 更强的注意力**, 而非 FFN 花样。
>
> 整条 K-Net→SGR 研究链至此彻底收敛: 论文秩洞察**造不出更好的 LLM 部件**——
> 描述性(rank=circuit complexity)成立, 处方性(rank-routing FFN)在真实 LM 指标下全面失败。
