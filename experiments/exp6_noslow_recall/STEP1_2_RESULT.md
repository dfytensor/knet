# Step1+2: 砍 SlowMemory(死代码) + 放大 GLA recall(真状态) — 完整结论

## Step 1: 砍掉 SlowMemory → ppl 不变(确认死代码)
| seq | vanilla(含SlowMem) | NoSlow(砍掉) | 差距 |
|-----|---------------------|--------------|------|
| 512 | 36.71 | 37.25 | 1.5%(噪声) |
| 4096 | 123.77 | 124.33 | 0.5%(噪声) |

gate=1.0 在所有长度(512-8192)→ SlowMemory 贡献 0% → **砍掉安全, ppl 不随 seq 变化**。

## Step 2: 放大 GLA recall(d_h 64→128, 4× 内容寻址状态) → 无可靠收益
| 配置 | recall 状态 | seq512 ppl | seq4096 ppl |
|------|------------|------------|-------------|
| vanilla (d_h=64) | 32768 | 37.28 | **78.82** |
| d_h=96 (2.25×) | 73728 | 37.32 | — |
| d_h=128 (4×) | 131072 | **36.92** | 79.91 |

seq512: d_h=128 好 1.0%(噪声内); seq4096: d_h=128 差 1.4%(反转)。**±1% 都在单种子噪声内, 无可靠方向。**

## 为什么放大真状态(GLA recall)也没用
1. **GLA recall 状态(32768 维)在 seq ≤ 4096 已充足** —— 内容寻址记忆够用, 非瓶颈。
2. **+1.3M 参数(d_h 投影)带来的增益被噪声淹没** —— 4× state 对 80M 模型微不足道。
3. 与 SlowMemory 不同, recall 确实被使用(32% 贡献), 但**"被使用" ≠ "是瓶颈"** —— 容量够用时, 放大有用部件也不提升。

## 完整回答: state 做大到底能带来什么
> **在本实验覆盖的规模(seq 512-8192, 80M 模型, 200-1500 步): state 做大什么都带不来。**
> - 死状态(SlowMemory): gate=1.0, 贡献 0%, 放大 = 放大 0。
> - 真状态(GLA recall): 贡献 32%, 但容量已充足, 4× 状态 = 噪声级效果。
> - 主状态(SSM 骨干): 贡献 35%, 同理预计也充足(未直接放大, 但 SlowMemory/Recall 都不缺, 主状态更不缺)。
>
> **state 容量不是 FRSMASH 的瓶颈。** 瓶颈是训练规模(步数 200→1500 = ppl 52→37, 30% 提升)和模型宽度(d512→d768 预计提升更大)。
> state 放大只有在 **seq >> 32000**(状态容量远不够)或 **d_h 极小**(recall 寻址能力不足)时才可能有用 —— 本设置都不满足。

## 一句话
> **state 做大在本规模带不来任何东西。** 死状态(SlowMemory)贡献 0%, 真状态(GLA recall)够用不缺。
> 提升路径不在 state 容量, 在**训练步数 + 模型宽度**。
